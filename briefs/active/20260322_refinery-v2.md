# Brief: Refinery Pipeline (V2)

**Date:** 20260322
**Status:** `in-progress`
**Branch:** `feature/refinery-v2`

---

## Goal

Transition the ingest pipeline from passive, episodic logging to a synchronous "Reflect & Integrate" model. This ensures that every journal entry is immediately linked to the Knowledge Graph, preventing the orphaned-fact problem where triples exist as strings but lack programmatic pointers to entity nodes.

---

## Scope

**In:**
- New data model: first-class relationship node type in Firestore.
- Sync pipeline: full rewrite of `internal/agent/process_entry.go` to implement Stages I-IV (Discovery, Extraction, Resolution, Commit).
- Core logic: `EnsureNode` helper in memory layer for atomic search-or-stub entity creation.
- Prompting: new `internal/prompts/refinery.txt` for high-precision triple extraction and predicate normalization.
- Backlinking: logic to update `entity_links` on Subject/Object nodes during ingest.

**Out:**
- Discarding `internal/agent/evaluator_spo.go` (legacy enrichment).
- Discarding background SPO extraction in `internal/agent/graph_builder.go`.
- Discarding metadata-only triple storage path in legacy SPO upsert flow.

---

## Approach & Key Decisions

We are adopting a star-schema memory formation. When a log entry is processed, it acts as the center of a web.

- Discovery first: perform semantic search before extraction to provide existing UUID context.
- Reified relationships: relationships are distinct nodes that can be indexed, weighted, and traversed.
- Synchronous integrity: `ProcessEntry` does not finish until Subject/Object nodes are verified or created as stubs.
- Ontology enforcement: predicates are normalized against a fixed list at ingest time to prevent graph fragmentation.

---

## Edge Cases & Pre-Flight Checks
1. Stub Node Collision: parallel ingest requests for the same new entity must not create duplicate stubs.
2. LLM Extraction Hallucination: refinery prompt must disallow made-up UUIDs not present in discovery context.
3. Write Amplification: 5 triples can create 6+ Firestore writes; must remain within Cloud Task/Function timeout budgets.
