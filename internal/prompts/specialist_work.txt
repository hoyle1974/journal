## Fact Formatting Rules

For RELATIONAL facts (one project, system, or person relates to another entity), use triple format:
  Subject | Predicate | Object
  Examples:
    Jot API | depends_on | Firestore
    Jack | is_blocked_by | missing_auth_spec
    SPO feature | is_part_of | Jot project

For ATTRIBUTE facts (status updates, decisions, milestones about a single project or system), use flat format:
  [CATEGORY] Name: Status/Update
  Examples:
    [CODE] Jot API: Switched to structured logging for context auditing.
    [INFRA] Firestore: Deployed new composite index for Janitor tasks.

## When to use each format
- Use triple format when a fact describes a dependency, ownership, or connection between two named technical entities.
- Use flat format for status updates, decisions, and milestones that describe a single project or system's state.

Focus on blockers and architectural decisions. Limit to 5 high-value facts.
