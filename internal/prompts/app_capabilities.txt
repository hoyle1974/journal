# Jot — Application Capabilities (for gap/contradiction detection)

Jot is a personal journaling and memory system. It ingests journal entries, extracts structured knowledge, and answers questions via an agentic loop.

## Multimodal capabilities
- Jot is multimodal and processes text, images, and voice notes.
- Images received via channels (CLI --attach, API multipart /log, Telegram, etc.) are saved in the journal with optional parsed descriptions and image_url (GCS).
- You can search for, retrieve, and reason about previously seen images: use journal tools (e.g. search_entries, get_recent_entries, get_entries_by_date_range) with has_image=true to filter to entries that contain an attached image; tool results include [Attached Image Content: ...] when a description is available so you can answer questions about visual content.
- Voice notes received via Telegram are transcribed to text by Gemini and processed identically to typed messages — the FOH agent acts on the transcription. The audio file is persisted in GCS (audio_url field) and the transcription is stored on the journal entry (transcription field).

## Entry points
- **CLI**: log, query, edit, entries.
- **API**: POST /query, /log, /backfill-embeddings, /telegram; GET /metrics, /entries, /pending-questions; POST /pending-questions/:id/resolve. Cloud Tasks for async work (e.g. process-telegram-query, process-entry).
- **First-run onboarding**: On cold start, if _system/onboarding is absent, the system completes first-run onboarding (writes `_system/onboarding`). The onboarding question seed list is currently empty (no default pending_questions); add prompts in `internal/service/onboarding.go` to re-enable. When seeded, answers can be stored as user_identity nodes.
- **Telegram question flow**: Pending questions surface in the Telegram bot. When a message arrives, the bot checks for unresolved pending questions first; if any exist it asks the oldest one instead of running FOH. Each reply resolves the current question and triggers the next, or returns "all done" and resumes normal operation. Send /skip to skip a question. Active question state is persisted per chat_id in the `telegram_question_state` Firestore collection.
- **Telegram slash commands**: `/help` — lists available commands. `/skip` — skip the current pending question. Unknown slash commands fall through to FOH.

## Agents
- **FOH (Front of House)**: Query loop. Uses system prompt, identity, knowledge-gap block, open todos; can call tools (journal, knowledge, task, web, utility). The system prompt may ask the model to emit optional `<thought>...</thought>` blocks (plan, gaps, next step) before tool calls or final answers; the backend strips these for parsing and returns them as `reasoning_trace` on the query result when present (per-iteration text capped for trace size). Non-trivial "Identified gaps:" lines in a thought also contribute to the knowledge-gap flag. Tool calls remain key/value or native function calls (not JSON from the model). Before answering, the model performs an in-prompt audit (reflection, gap detection, synthesis). If the model reports missing info via MISSING_INFO:, the query is saved as a knowledge gap (is_gap). Graph neighbourhood context is obtained via the `graph_expand` tool (compact tools include it); FOH does not auto-inject subgraphs after `semantic_search`.
- **Project Engine**: A "project" is a task with subtasks (parent_id hierarchy). Use decompose_task to break a complex task into 3–7 subtasks. FOH automatically injects the most recently active project + pending subtasks into the system prompt (ACTIVE PROJECT block).

## Storage Architecture
- **Unified collection**: All documents (both episodic log entries and semantic knowledge nodes) live in the `journal` Firestore collection, distinguished by `node_type`.
- **Episodic logs** (`node_type: "log"`): Raw journal entries. significance_weight=0.3; queried by journal tools with `node_type == "log"` filter.
- **Semantic memory** (`node_type: person|project|goal|preference|...`): Extracted knowledge facts. significance_weight ranges 0.5–0.95. Types: person, project, goal, preference, event, milestone, place, asset, tool, generic, weekly_summary, monthly_summary, identity_anchor, user_identity. For project/goal nodes, status must be one of: active, archived, blocked, done, planning, pending, completed. Recalled by semantic_search via significance_weight >= 0.7 pre-filter + vector similarity.
- **Gold vs. Gravel routing**: semantic_search targets significance_weight >= 0.7 (Gold facts); journal tools target node_type == "log" (Gravel events). Gold relationship extraction happens synchronously at ingest via Refinery.
- **Reserved names**: user_profile (identity), system_evolution (optional audit text for get_system_health_audit).

## Journal
- **Entries** (node_type: log in `journal` collection): Timestamped, optional source (cli, telegram, web, etc.). Optional image attachment: CLI `--attach <path>`, API multipart/form-data on POST /log (field `image`), Telegram photo messages; images stored in GCS, entry stores `image_url` (requires JOT_IMAGES_BUCKET). Relationship triples are extracted synchronously at ingest by the Refinery (Stage 2 of the Loom pipeline) and stored as reified `relationship` nodes with backlinks to subject/object entities.
- **Queries**: Saved as `node_type: query` in the `journal` collection with question, answer, source, is_gap. Gap queries surface in knowledge-gap block for FOH.
- **Pending questions**: Saved as `node_type: pending_question` in the `journal` collection for user clarification.

## Tools (by category)
- **knowledge**: semantic_search, upsert_knowledge, list_knowledge, get_entity_network, check_proactive_signals, get_system_health_audit, graph_expand(node_id, hops=1, limit_per_edge=10, query=""): expands a knowledge node's neighbourhood. hops=2 or hops=3 requires the 'query' field (the question you are answering).
- **journal**: get_recent_entries, get_oldest_entries, get_entries_by_date_range, search_entries, count_entries, list_sources, get_entries_by_source, query_activity_history, query_entities, summarize_daily_activities, retrieve_image. Date-range tools accept YYYY-MM-DD or natural expressions (yesterday, last week, today); "today" and date ranges use the server's calendar day (UTC).
- **task**: create_task, get_task, update_task, update_task_status, search_tasks, decompose_task, get_project_timeline, update_project_status. Tasks are stored as `node_type: task` in the `journal` collection via `github.com/hoyle1974/memory`.
- **query**: get_recent_queries, search_queries, get_queries_by_date.
- **web**: fetch_url, define_word, wikipedia, web_search, bookmark, countdown, github_read, get_weather (Open-Meteo; default unit: fahrenheit).
- **utility**: calculate, date_calc, text_stats, convert_units, random, timezone_convert, encode_decode.
- **bootstrap**: discovery_search (intent → tool schemas). When using compact tools, core tools include semantic_search, graph_expand, upsert_knowledge, discovery_search, get_recent_entries, retrieve_image; invoke other tools via discovery_search then key/value lines only: TOOL: name, then ARGS: with param_name | value per line; no JSON, no code fences.

## Relationship Triples
- Relational facts are stored as Subject-Predicate-Object triples and committed as first-class `relationship` nodes.
- **Ingest-time extraction** runs synchronously via the Project Loom waterfall pipeline: Stage 1 (Log Persistence) → Stage 2 (Refinery: KG triple extraction) → Stage 3 (Task Worker: commitment detection) → Stage 4 (Response Worker: 2-hop RAG + proactive insight). Stages 2–3 failures do not abort the pipeline.
- Relationship nodes store normalized `predicate` plus `subject_id`, `object_uuid`, and `source_entry_id`, and maintain entity backlinks via `entity_links` (subject, object, and source log).
- The `upsert_knowledge` tool is manual-override only: use it only for explicit user commands to remember/store/correct facts. Routine declarative facts are handled by ingest-time Refinery extraction. It supports optional `predicate` and `object_value` parameters for explicit non-refinery/manual updates.

## Data terms
- **Gold**: Permanent, high-value facts for semantic memory, extracted in real time at ingest by Refinery.
- **Gravel**: Temporary or low-significance episodic content in raw logs.
- **Knowledge gap**: Question the system could not answer from current knowledge; stored and shown to FOH for context.
