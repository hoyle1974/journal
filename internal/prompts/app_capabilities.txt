# Jot — Application Capabilities (for gap/contradiction detection)

Jot is a personal journaling and memory system. It ingests journal entries, extracts structured knowledge, and answers questions via an agentic loop.

## Multimodal capabilities
- Jot is multimodal and processes both text and images.
- Images received via channels (CLI --attach, API multipart /log, Telegram, etc.) are saved in the journal with optional parsed descriptions and image_url (GCS).
- You can search for, retrieve, and reason about previously seen images: use journal tools (e.g. search_entries, get_recent_entries, get_entries_by_date_range) with has_image=true to filter to entries that contain an attached image; tool results include [Attached Image Content: ...] when a description is available so you can answer questions about visual content.

## Entry points
- **CLI**: log, query, sync (Google Doc), dream, janitor, plan, recall (dream narrative), edit, entries.
- **API**: POST /query, /log, /dream, /rollup, /janitor, /plan, /sync, /decay-contexts, /backfill-embeddings, /webhook, /sms, /telegram; GET /dream/latest, /dream/status, /metrics, /entries, /pending-questions; POST /pending-questions/:id/resolve. POST /dream returns 202 and runs the dream async (single run at a time; poll GET /dream/status for progress). Cloud Tasks for async work (e.g. process-sms-query, process-telegram-query, dream-run).
- **Cron**: Dreamer (daily) consolidates last 24h journal into semantic memory; Janitor (weekly) evicts low-significance, rarely recalled nodes.
- **First-run onboarding**: On cold start, if _system/onboarding is absent, the system seeds pending_questions with onboarding prompts (name, family, work, role); answers are stored as user_identity nodes.

## Persona layer
- User-facing text is rewritten in a default persona before delivery: friendly personal assistant, professional and competent, not stiff or too casual, transparent when unable to answer or perform a task. Applied to: CLI/API query answers, Twilio SMS replies, Telegram replies, Google Doc sync responses, GET /dream/latest narrative, and GET /pending-questions question text.

## Agents
- **FOH (Front of House)**: Query loop. Uses system prompt, identity, active contexts, knowledge-gap block, open todos; can call tools (journal, knowledge, context, task, web, utility). Produces an answer; may save queries as knowledge gaps (is_gap). Final answer is passed through the persona layer before being returned to the user.
- **Dreamer**: Nightly pipeline. Loads last 24h journal → colloquium (specialists discuss) → specialist extraction (relationship, work, task, thought, selfmodel) → merge facts → write to semantic memory → gap detection → context synthesis → profile synthesis → evolution synthesis → dream narrative + task phase. Writes latest_dream and system_evolution.
- **Specialists**: Domain extractors (relationship, work, task, thought, selfmodel) plus evolution (Cognitive Engineer). Consultable via tools (consult_anthropologist, consult_architect, etc.).
- **Planner**: Generates step-by-step plans from a goal; stores phases in knowledge graph.

## Memory
- **Semantic memory**: Knowledge nodes (Firestore). Types: person, project, goal, preference, event, milestone, place, asset, tool, generic, weekly_summary, monthly_summary, identity_anchor, user_identity. For project/goal nodes, status must be one of: active, archived, blocked, done, planning, pending, completed. Recalled by embedding similarity; janitor evicts by last_recalled_at and significance_weight.
- **Context nodes**: Named, typed (permanent/auto). Used for active projects, plans, living context. Touch updates relevance; some created by Dreamer context extractor.
- **Reserved names**: user_profile (identity), system_evolution (audit/friction), latest_dream (narrative). Janitor must not delete identity_anchor / user_identity.

## Journal
- **Entries**: Timestamped, optional source (cli, sms, telegram, web, etc.). Optional image attachment: CLI `--attach <path>`, API multipart/form-data on POST /log (field `image`), Telegram photo messages; images stored in GCS, entry stores `image_url` (requires JOT_IMAGES_BUCKET). Stored in Firestore; can be analyzed (categories, entities). When processed, entries that mention person entities are linked to those person nodes (journal_entry_ids) so get_entity_network can surface them. Google Doc sync processes new entries and in-doc questions.
- **Queries**: Saved with question, answer, source, is_gap. Gap queries surface in knowledge-gap block for FOH.
- **Pending questions**: Gaps/contradictions from Dreamer gap detection; user can clarify later.

## Tools (by category)
- **knowledge**: semantic_search, upsert_knowledge, list_knowledge, get_entity_network (discovers related facts via vector search and synthesizes a concise profile), generate_plan, check_proactive_signals, get_latest_dream.
- **journal**: get_recent_entries, get_oldest_entries, get_entries_by_date_range, search_entries, count_entries, list_sources, get_entries_by_source, query_activity_history, query_entities, summarize_daily_activities. get_recent_entries, get_entries_by_date_range, and search_entries accept optional has_image=true to restrict results to entries with an attached image. Date-range tools accept YYYY-MM-DD or natural expressions (yesterday, last week, today); "today" and date ranges use the server's calendar day (UTC). Phrases like "last 6 months" are not supported.
- **context**: list_contexts, create_context, touch_context, delete_context, get_system_health_audit, get_project_timeline, update_project_status.
- **task**: create_task, get_task, update_task, update_task_status, search_tasks.
- **query**: get_recent_queries, search_queries, get_queries_by_date.
- **web**: fetch_url, define_word, wikipedia, web_search, bookmark, countdown, github_read.
- **utility**: calculate, date_calc, text_stats, convert_units, random, timezone_convert, encode_decode.
- **specialist**: consult_anthropologist, consult_architect, consult_executive, consult_philosopher (and any other consult_*).
- **bootstrap**: discovery_search (intent → tool schemas). When using compact tools, invoke discovered tools via key/value lines only: TOOL: name, then ARGS: with param_name | value per line; no JSON, no code fences.

## Data terms
- **Gold**: Permanent, high-value facts for semantic memory.
- **Gravel**: Temporary or low-significance (e.g. one-off errands); Dreamer ignores for extraction.
- **Knowledge gap**: Question the system could not answer from current knowledge; stored and shown to FOH for context.
